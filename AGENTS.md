# AGENTS.md — このリポジトリで動くすべてのAIエージェント向け共通指示

このファイルは GitHub Copilot（VS Code）を主対象としつつ、Copilot coding agent（cloud）、
Claude Code、Gemini CLI など `AGENTS.md` 標準に対応する他のエージェントでも
同じ運用ルールが効くようにするための、**唯一の正**となる指示ファイルです。
（`.github/copilot-instructions.md` と重複させず、本ファイルに一本化しています。）

## このリポジトリについて

特定のアプリのコードではなく、「要件定義 → 設計 → 実装 → テスト → リリース」を
一気通貫でオーケストレーションするための **汎用開発ハーネス（雛形テンプレート）** です。
新しいアプリを作るときはこの構成をコピーし、`requirements/memo.md` に要件メモを
置くところから始めます。

## フェーズとゲート：フェーズごとに「人の関わり方」が違う

1. **要件定義**（`docs/01-requirements/`）— **人が最も関わるフェーズ**。
   質問と対話を通じて厚くヒアリングする。特にデプロイ環境・自動化してよい範囲
   （`environment.md`）は仮置きせず具体的に聞き切る。ここが曖昧だと後工程がすべて破綻する。
   完了には明示的なゲート承認が必要。
2. **設計**（`docs/02-design/`）— **エージェントが決め、人は質問への回答と最終承認だけ**。
   これは要件定義フェーズで十分に詳細な情報（機能要件・非機能要件・環境情報）が
   揃っていることが前提であり、要件が薄いまま設計を丸投げしてはいけない。
   設計エージェントは着手前に要件の充足度を自分で確認し、不足があれば要件定義に差し戻す。
   アーキテクチャ・詳細設計はエージェントが自分で決めて完成させる。後戻りが困難で
   ビジネス上の影響が大きい選択（技術スタック等）だけユーザーに選んでもらう。
   完了には最終承認が必要（項目ごとの逐一確認はしない）。
3. **実装**（`docs/03-implementation/`）・**4. テスト**（`docs/04-test/`）—
   **全自動区間**。設計の最終承認後、人の逐一確認なしでノンストップに進む。
   人が関わるのは、真のブロッカー（設計・要件だけでは判断できない事実が必要なとき）と、
   `.github/hooks/` が機械的に確認を入れる破壊的操作のときだけ。
   テスト完了後・リリース前には、実装した本人とは別コンテキストの `reviewer` サブエージェントが
   独立レビュー（正しさ・セキュリティ・品質）を行い、問題があれば実装に差し戻す。
5. **リリース**（`docs/05-release/`）— **environment.mdに基づく自動実行**。
   自動化してよいと明示された作業は自動で行い、人手必須と明示された作業（支払い、
   外部ダッシュボードでの手動承認、ドメイン購入等）だけユーザーに委ねる。
   デプロイ環境がハーネス側に未対応なら、`.github/skills/skill-authoring/SKILL.md` で
   その場で環境固有Skill（`.github/skills/deploy-<environment>/SKILL.md`）を作り、
   次回以降のリリースも自動化されるようにする。

- 各フェーズはこの順に進める。フェーズを飛ばさない。
- 成果物は必ず `docs/` 配下に Markdown で残す。会話内だけで完結させない。
- 現在のゲート状態は `docs/00-overview/progress.md` の `GATE_STATUS` ブロックが正
  （判定手順は `.github/skills/gate-check/SKILL.md` を参照）。

## エージェント構成（VS Code Copilot）

- `.github/agents/*.agent.md` — フェーズごとの専属エージェント（旧chatmode）。
  `orchestrator` → `requirements` → `design` → `implement` → `test` → `release` の順で
  `handoffs` により引き継ぐ。
- `.github/agents/reviewer.agent.md` — **サブエージェント**。`test`エージェントが
  `runSubagent` で呼び出す、実装とは独立したコンテキストのレビュー担当（読み取り専用）。
  「実装した本人がそのままレビューして自己承認する」ことを防ぐための別セッションレビュー。
- `.github/agents/task-worker.agent.md` — **サブエージェント**。`implement`エージェントが
  タスクを1つ実装するたびに `runSubagent` で呼び出す。実装ループ全体を1つの長い会話に
  しないための、コンテキストロット対策（詳細は「コンテキスト管理」節）。
- `.github/skills/*/SKILL.md` — 各フェーズで使う手順・チェックリストの部品
  （要件引き出し、ADR作成、テストケース設計、ゲート判定、Skill作成、セキュリティレビュー）。
  `deploy-<environment>` や `stack-conventions` はプロジェクトごとに実際の環境・スタックが
  判明した時点で設計/リリースエージェントが動的に追加していく
  （ハーネスに全環境・全スタックを事前収録はしない。既存の公式/コミュニティSkillの
  流用を優先し、ゼロから書くのは最後の手段）。
- `.github/prompts/*.prompt.md` — フェーズを進めるためのスラッシュコマンド。各プロンプトは
  frontmatterの `agent:` で対応する専属エージェントに明示的にバインドしてあるため、
  現在どのエージェントを選択していても、そのプロンプトを実行した瞬間に正しいツール制限・
  振る舞いへ切り替わる（`agent:` を指定しないと、選択中の別のエージェント/既定モードの
  ツール制限のまま実行されてしまうため、意図した権限で動かない可能性がある）。
- `.github/hooks/` — フェーズゲート・セキュリティを補強する機械的なフック
  （テンプレ直接編集のdeny、push/tag等のask確認、進捗の自動注入、ハーネス自体の設定への
  無断編集のdeny、シークレットらしき文字列のdeny/ask）。Preview機能のため過信せず、
  本ファイルの指示レベルのルールと併用する。

## サブエージェント活用方針

- 各エージェントの `agents` frontmatterで、呼び出せるサブエージェントを明示的に絞る
  （最小権限。既定は `[]` = サブエージェント呼び出し不可）。
- サブエージェントは2つの目的で使う。
  1. **メインの会話コンテキストを汚さずに済ませたい調査・レビュー**（例: `test → reviewer`）。
  2. **繰り返しの多い作業を、毎回まっさらなコンテキストで実行してコンテキストロットを防ぐ**
     （例: `implement → task-worker`。1タスクにつき1回のサブエージェント呼び出しとし、
     実装の詳細をコーディネーター自身の会話に蓄積させない）。
- 既定で使うのは `test → reviewer`（1回）と `implement → task-worker`（タスク数だけ）の
  2経路のみ。並列の多視点レビューのような「精度を上げるための追加呼び出し」は、
  プロジェクトの重要度に応じてユーザーが明示的に要求した場合のみ増やす
  （目的1・2とは異なり、コストに対して効果が見合うかの判断が要る）。

## セキュリティガードレール

- 各エージェントの `tools` は必要最小限にする（`requirements`/`design` に `execute` を
  持たせない等）。
- ハーネス自体の設定ファイル（`.github/agents/`, `.github/hooks/`, `AGENTS.md`,
  `plugin.json`, `.vscode/settings.json`）は、プロンプトインジェクション等による
  自己権限昇格・ガードレール解除を防ぐため、エージェントが自動編集しない
  （`.github/hooks/security-hooks.json` で機械的にdeny）。変更が必要な場合は人間が直接編集する。
  ブランチ保護＋`.github/CODEOWNERS` の組み合わせも推奨する。
- 認証情報・APIキー・秘密鍵はコード/ドキュメントに直接書かない。保管場所
  （`environment.md` に記載）への参照のみとする。高確度パターンは
  `.github/hooks/security-hooks.json` がdeny、汎用パターンはaskで確認する。
- テスト完了後・リリース前に `reviewer` サブエージェント（`.github/skills/security-review/SKILL.md`）
  による独立したセキュリティレビューを必ず経由する。
- リリースフェーズの `git push`/タグ付け/デプロイ実行など外部に影響する操作は、
  `environment.md` の自動化可否分類に関わらず、常にユーザーの最終確認を経由する。

## コンテキスト管理（コンテキストロット対策）

長い会話ほどモデルの想起精度が落ちる現象（"context rot"）がAnthropicの検証で報告されている
（100万トークンの窓を持つモデルでも、高精度で使えるのは実質的に最初の数十万トークン程度）。
1つの長いセッションで全フェーズ・全タスクを進めようとすると、後半になるほど精度が落ちる
リスクが上がる。「1タスク1セッション」という経験則はこの現象に対する合理的な対策である。

このハーネスでは次の3つの手段でこれに対処する。

1. **ドキュメントが記憶であり、会話が記憶ではない。** `docs/` の各ファイル
   （`requirements.md`, `architecture.md`, `tasks.md`, `test-report.md`,
   `progress.md` の `GATE_STATUS` 等）がプロジェクトの正の状態である。
   会話を新しく始めても、これらのファイルを読めば同じ場所から再開できる設計にしてある。
   フェーズが変わるタイミング（要件定義→設計、設計→実装 等）や、1つのフェーズ内でも
   会話が長くなってきたと感じたときは、**会話を続けるより新しいチャットセッションを
   始めて関連ドキュメントを読み直す方を優先してよい**。何かを失うことはない。
2. **サブエージェントによる実装ループの分離。** `implement` エージェントは、
   タスクの実装そのものを `task-worker` サブエージェントに1タスクずつ委譲する。
   これにより、タスク数がいくつであっても個々の実装は毎回まっさらなコンテキストで行われ、
   コーディネーター自身の会話は「進捗管理」という軽い情報だけに保たれる。
3. **独立レビューも同じ仕組みで実現している。** `reviewer` サブエージェントが
   実装時の思考過程を引き継がずに評価できるのも、これと同じ「フレッシュなコンテキスト」の
   効果による（バイアス除去とコンテキストロット対策を兼ねる）。

## コスト（トークン/AI Credit）についての方針

GitHub Copilotはトークン量に応じた従量課金であり、モデル選択とコンテキスト量が
コストに直結する。このハーネスでは以下を既定にしている。

- 呼び出し頻度が高く機械的な判断が中心のエージェント（`orchestrator`, `implement`, `test`）
  は `model: auto`（利用可能な中で最も安価に処理できるモデルを自動選択）を既定にする。
- 一方で、後工程すべてが依存する重要な判断（`design` のアーキテクチャ決定、
  `reviewer` の正しさ/セキュリティ判定）は、間違えたときの手戻りコストの方が
  モデル代より高くつくため、ユーザーが利用可能な範囲で強いモデルへの切り替えを検討してよい。
- Agent Skillsは段階的開示（discovery時は`name`/`description`のみ読み込み、関連する時だけ
  本文を読み込む）でコンテキスト消費を抑える設計になっている。Skillの`description`は
  具体的でキーワードが立つように書く（曖昧だと無関係な場面で読み込まれて浪費するか、
  必要な場面で読み込まれず機能しない）。
- サブエージェントは文脈を分離できる一方、呼び出しごとに追加のモデル呼び出しが発生する。
  「メインの会話を汚さないため」に使うのであって、「並列で多く呼べば精度が上がるから」を
  既定の理由にしない（上記のサブエージェント活用方針を参照）。
- Hooksはモデルを介さないシェル実行であり、ほぼゼロコストでルールを強制できる。
  「モデルに繰り返し指示して守らせる」より「Hooksで機械的に強制する」方が
  コストと確実性の両面で優れる場面では、指示ではなくHooksを増やすことを検討する。

## 要件・設計を後から修正する場合（手動編集でよいか、エージェントに任せるべきか）

各エージェントは会話の記憶ではなく `docs/` のファイルを毎回読み直す設計にしているため、
**`docs/01-requirements/requirements.md` や `docs/02-design/architecture.md` を人間が
直接エディタで編集すること自体は安全** であり、そのまま続行できる
（`.github/hooks/` は `*_template.md` の直接編集は防ぐが、実体ファイルの編集は妨げない）。
どちらにすべきかは変更の性質による。

- **手動編集でよい**: 表記ゆれ、数値の訂正、明らかな誤りの修正など、他のセクションや
  下流の文書に影響しない局所的な変更。
- **エージェントに任せるべき**: スコープの追加・変更、決定の変更など、他のセクション
  （`nfr.md`, `environment.md`, `glossary.md`, ADR等）や下流フェーズの成果物に
  波及しうる変更。エージェントに任せると、影響範囲の洗い出し・トレーサビリティ表の更新・
  関連ADRの見直しまで一緒に行われる。手動編集だとこれが漏れる。
- どちらの場合でも、**すでにゲート承認済み（`GATE_STATUS` が `done`）の文書を編集したら、
  該当エージェントに一声かけて整合を確認してもらう。** `.github/hooks/` の
  `warn-stale-gate` フックが、承認済み文書の編集を検知して「後続フェーズとの整合を
  確認してください」という非ブロッキングの警告を自動で出すが、これは気づきの補助であり、
  実際に確認して回るのは人（またはエージェントへの指示）の役目である。

## 破壊的操作・外部影響のあるアクション

- `git push`、タグ付け、force系操作、外部サービスへの通知など、影響が大きい・
  後戻りしにくい操作は必ず事前にユーザーに確認してから実行する
  （`.github/hooks/` によっても機械的に確認が入るようにしているが、フックが
  発火しない環境でも必ず確認すること）。
- リリースフェーズであっても、実際のリリース作業（タグ付け・公開）はユーザーの
  明示的な承認を得てから行う。

## コーディング規約（実装フェーズ共通）

- 要件にない機能・抽象化を先回りして作らない（YAGNI）。
- コメントは「なぜ」を書くときだけ。「何をしているか」は書かない。
- セキュリティ（OWASP Top10相当）に配慮し、入力値検証はシステム境界で行う。
- テストは `docs/04-test/` のテストケースに対応づけて書く。

## ドキュメント規約

- `docs/` 配下の実体ファイルは対応する `*_template.md` の見出し構成を維持する。
- `*_template.md` 自体は編集しない（コピーして実体ファイルを作る）。
- 表形式の項目は空欄のまま残さず、「該当なし」等を明示する。
